# Architecture Analysis

## 1. 프로젝트 요약

이 프로젝트는 Java Swing 기반의 타자 게임 클라이언트와 소켓 기반 멀티플레이 서버로 구성된다.

- 서버 진입점: `src/server/ServerMain.java`
- 서버 핵심: `src/server/GameServer.java`
- 클라이언트 진입점: `src/client/app/ClientMain.java`
- 클라이언트 네트워크/이벤트 허브: `src/client/app/GameClient.java`, `src/client/network/MessageHandler.java`
- 인게임 로직: `src/server/game/*`, `src/client/ui/game/*`
- 공용 모델: `src/game/model/*`

## 2. 런타임 구조

### 서버 측

1. `ServerMain`이 `GameServer`를 생성하고 포트를 연다.
2. `GameServer`는 클라이언트 연결마다 `ClientHandler` 스레드를 생성한다.
3. `ClientHandler`는 한 줄 단위 메시지를 읽고 `GameServer` 또는 `ServerGameController`에 위임한다.
4. 방이 시작되면 `ServerGameController`가 생성되고 단어 생성 스케줄러와 pH 체크 스케줄러를 실행한다.

### 클라이언트 측

1. `ClientMain`이 로그인 정보를 받아 `GameClient`를 생성한다.
2. `GameClient`는 소켓을 열고 수신 스레드에서 `MessageHandler`로 서버 메시지를 넘긴다.
3. `MessageHandler`는 문자열 메시지를 이벤트로 변환한다.
4. 현재 화면 객체(`MainMenu`, `RoomListDialog`, `GameLobby`, `GameScreen`, `LeaderboardDialog`)가 단 하나의 이벤트 리스너로 등록되어 이벤트를 받는다.

## 3. 현재 설계의 장점

- 수업 프로젝트로서는 흐름이 단순하고 읽기 쉽다.
- 서버와 클라이언트가 각각 역할별 클래스로 분리돼 있다.
- `GameEvent`에 프로토콜 상수가 어느 정도 모여 있어 문자열 난립을 완화하려는 시도가 보인다.
- 리더보드, 단어 관리자, 게임 상태를 별도 클래스로 분리한 점은 확장 여지가 있다.

## 4. 핵심 구조적 문제

### 4.1 프로토콜과 구현이 강하게 결합됨

현재 통신은 `|`, `,`, `;` 구분자에 의존한다. 이 방식은 구현은 빠르지만 다음 문제가 있다.

- 필드 이스케이프가 없다.
- 명세보다 구현 우선으로 흘러서 메시지 타입이 혼재한다.
- 클라이언트/서버 어느 한쪽만 바뀌어도 쉽게 깨진다.
- `PLAYER_LIST_RESPONSE`처럼 상수는 있지만 실제로는 거의 사용되지 않는 메시지가 생긴다.

### 4.2 단일 이벤트 리스너 구조

`GameClient`는 한 번에 하나의 `GameEventListener`만 가진다.

- 화면 전환마다 `setEventListener(...)`를 덮어쓴다.
- 팝업/다이얼로그가 열릴 때 다른 화면 이벤트를 놓칠 수 있다.
- UI 전환 로직과 네트워크 이벤트 라우팅이 섞인다.

### 4.3 서버 상태와 UI 상태가 동시에 사실상 진실 소스 역할을 함

예를 들어 `GameLobby`와 `GameScreen`은 서버 메시지를 받으면서도 로컬 객체 `GameRoom`, `activeWords`, `myScore`를 자체적으로 수정한다.

- 서버 권위(authoritative state)와 UI 캐시의 경계가 불명확하다.
- 방 플레이어 수, 방장 여부, 게임 종료 후 상태 복구 같은 부분에서 불일치가 생기기 쉽다.

### 4.4 2인 전용 게임 로직과 4인 방 옵션 충돌

코드 곳곳은 사실상 1:1 대전을 가정한다.

- `GameRoom`은 최대 4인까지 허용한다.
- `ServerGameState.getOpponentOf()`는 정확히 2명이라고 가정한다.
- `GameScreen`은 상대 점수 라벨도 하나만 가진다.

즉, 현재 도메인 모델은 2인용과 다인용이 동시에 존재하는데, 실제 구현은 2인 전용에 가깝다.

## 5. 리팩터링 우선순위

### 단계 1: 규칙 고정

먼저 명세를 하나로 고정해야 한다.

- 게임은 `2인 전용`으로 제한할지
- 아니면 `3~4인 지원`으로 진짜 확장할지
- 프로토콜은 기존 문자열 포맷을 유지할지
- 아니면 JSON line 기반으로 바꿀지

현재 코드 상태에서는 `2인 전용으로 명세를 먼저 고정`하는 것이 가장 현실적이다.

### 단계 2: 프로토콜 계층 정리

- 요청/응답/이벤트를 분리한다.
- `ClientCommand`, `ServerMessage`, `ClientEvent`의 책임을 다시 나눈다.
- 문자열 직접 조립 대신 메시지 DTO 또는 최소한 빌더/파서 클래스를 둔다.

### 단계 3: 상태 관리 정리

- 서버를 권위 상태의 단일 소스로 둔다.
- 클라이언트는 서버 상태 스냅샷을 렌더링하는 쪽으로 단순화한다.
- 방 상태와 게임 상태를 명확히 분리한다.

### 단계 4: UI 이벤트 구조 개선

- `GameClient`에 다중 리스너 또는 화면별 구독 해제를 지원한다.
- 또는 중앙 이벤트 버스를 두고 화면은 필요한 이벤트만 수신하게 바꾼다.

## 6. 권장 리팩터링 순서

1. 최대 인원을 2명으로 제한하거나, 반대로 다인전 로직을 완성한다.
2. 게임 종료 시 방 상태를 정상 복구한다.
3. 프로토콜 스펙 문서를 기준으로 상수/메시지 타입을 정리한다.
4. 문자열 분해 로직을 전용 파서로 추출한다.
5. `GameClient` 단일 리스너 구조를 개선한다.
6. UI가 서버 상태를 추정하지 않도록 로컬 상태 갱신 코드를 줄인다.

## 7. 추천 패키지 재구성

현 구조를 크게 깨지 않으면서 다음 정도로 나누면 관리가 쉬워진다.

- `protocol`
  - 메시지 타입, 요청/응답 모델, 직렬화/역직렬화
- `server.room`
  - 방 생성/입장/퇴장/호스트 위임
- `server.game`
  - 현재와 유사하되 2인/다인 규칙을 명확히 분리
- `client.session`
  - 네트워크 연결, 이벤트 브로커, 현재 세션 상태
- `client.ui.*`
  - 순수 UI 렌더링과 사용자 입력 처리

## 8. 검증 메모

정적 분석과 컴파일 검증을 수행했다.

- 수행 명령: `javac $(find src -name '*.java')`
- 결과: 컴파일 성공, varargs 관련 경고 3건

런타임 실플레이 테스트는 이번 작업 범위에 포함하지 않았다.
